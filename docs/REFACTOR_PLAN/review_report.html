2.2 竞品差异化不足 ​

问题: 市面上已有类似产品，计划缺乏明确差异化策略。

竞品分析:

  4. 深度评估: 活跃度、难度、适用场景等 AI 评估维度

3.2 数据加载策略缺失 ​

问题: projects.json 约 15MB，topics.json 约 9MB，原计划未说明前端加载策略。

分析:

当前数据规模:

  • projects.json: 14.9MB (7,661 项目)
  • topics.json: 9.2MB
  • 总计: ~24MB JSON 数据

问题:

  • 首次加载 24MB → 移动端不可接受
  • 全量数据加载 → 内存压力

建议数据分层策略:

┌─────────────────────────────────────────────────────────────┐
│                    数据分层加载                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Layer 1: 轻量索引 (~500KB)                                 │
│  ────────────────────────────────────────────────────────  │
│  文件: data/index-lite.json                                 │
│  内容: {name, stars, category, subcategory, health}         │
│  用途: 首屏展示、快速筛选                                    │
│                                                             │
│  Layer 2: 分类切片 (~1-2MB per category)                    │
│  ────────────────────────────────────────────────────────  │
│  文件: data/categories/agents.json                          │
│  内容: 当前分类完整项目数据                                  │
│  用途: 进入分类时懒加载                                      │
│                                                             │
│  Layer 3: 项目详情 (~10KB per project)                      │
│  ────────────────────────────────────────────────────────  │
│  文件: data/projects/autogpt.json                           │
│  内容: 单项目完整信息 + 相关推荐                             │
│  用途: 详情页按需加载                                        │
│                                                             │
│  Layer 4: 聚合数据 (~200KB)                                  │
│  ────────────────────────────────────────────────────────  │
│  文件: data/aggregations.json                               │
│  内容: 统计、趋势、热门                                      │
│  用途: 首页、趋势报告                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘

预计算脚本建议:

bash
# 构建时生成分层数据

输出:
  data/index-lite.json        # 500KB
  data/categories/            # 13 个切片文件
  data/projects/              # 按需详情文件 (可选)
  data/aggregations.json      # 聚合统计

3.3 搜索方案缺失 ​

问题: 原计划建议 FlexSearch，但未分析 7,661 项目全文搜索的可行性。

分析:

搜索方案优点缺点适用性
FlexSearch轻量、前端运行7661项目内存压力大、中文支持弱❌
MiniSearch更轻量、支持中文分词需要自定义分词⚠️
MeiliSearch高性能、中文支持好需后端服务✅ (推荐)
Algolia最优体验收费、数据隐私⚠️

推荐方案: 构建时预生成搜索索引

javascript
// 构建时生成搜索索引 (静态JSON)
// scripts/generate-search-index.js

const searchIndex = projects.map(p => ({
  id: p.name,
  name: p.name,
  description: p.description,
  tags: p.tags,
  category: p.category,
  // 预计算搜索权重
  _score: p.stars / 1000 + (p.health === 'Active' ? 10 : 0)
}));

// 输出: data/search-index.json (~2MB)

前端搜索实现:

  • 加载搜索索引 (~2MB，可接受)
  • 使用 MiniSearch 或 Fuse.js 前端搜索
  • 中文分词使用预计算的关键词索引
